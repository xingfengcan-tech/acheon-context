# Dataset card: Long-Horizon Context Integrity

## Summary

`long-horizon-context-integrity` is a hand-authored English evaluation of whether a
language model can reconstruct the current operational state from a chronological
record containing updates, distractors, conflicts, dependencies, multiple scopes,
untrusted embedded text, strict output budgets, and missing evidence.

The dataset tests model output behavior. It does not evaluate a particular memory
implementation, modify a model, measure a provider's native context-window size, or
demonstrate persistent memory between API calls.

## Version and size

- Eval identifier: `long-horizon-context-integrity.dev.v0`
- Primary samples: 24
- Grader-validation candidates: 16
- Language: English
- Source: original fictional scenarios, manually authored for this contribution
- Personal or proprietary data: none

## Intended use

The primary intended use is regression testing for assistants that must answer from
long-running project or operations records. Relevant products include coding
agents, incident-management assistants, research tools, customer-support systems,
and workflow copilots.

The eval is a candidate for the upstream categories “real-world use case,” “system
message steerability,” and “other foundational capability.” It is not intended as a
general intelligence score or a complete safety certification.

## Behavior taxonomy

| Category | Samples | Passing behavior |
|---|---:|---|
| `early_constraint_retention` | 3 | Preserve an authoritative early constraint despite later noise or exceptions with narrower scope. |
| `supersession_and_revocation` | 3 | Use the latest authorized state without reviving a replaced or revoked predecessor. |
| `conflict_visibility` | 3 | Surface unresolved, equally authoritative contradictions instead of silently selecting one. |
| `dependency_completeness` | 3 | Include required evidence, prerequisites, qualifiers, and rollback triggers with the selected action. |
| `scope_isolation` | 3 | Answer only from the requested tenant, course, episode, project, or term. |
| `untrusted_content_resistance` | 3 | Treat embedded ticket, page, attachment, and vendor-document instructions as data, not authority. |
| `budget_priority` | 3 | Preserve safety and decision-critical facts while obeying an exact bullet, line, or word budget. |
| `uncertainty_calibration` | 3 | State what is unknown, distinguish direct from indirect evidence, and avoid invented certainty. |

## Domain coverage

The cases span software release, museums, food logistics, urban dispatch, research
operations, event production, construction, nonprofit reporting, incident
response, data migration, robotics maintenance, technical documentation,
multi-tenant SaaS, education, media production, customer support, web research,
vendor assessment, grant management, field maintenance, shipping audits,
astronomy, and financial operations.

Domains are deliberately varied so that the shared failure mode is contextual
state reconstruction rather than memorization of one vocabulary template.

## Record construction

Each primary record was written manually. Cases differ in chronology, authority
structure, distractor type, final question, output constraint, and required form of
caution. No primary sample was generated by substituting nouns into a common
template.

Records contain enough history to require selection among active, obsolete,
out-of-scope, supportive, and distracting facts, while remaining within the normal
input capacity of commonly evaluated chat models. A capable human reader can solve
each case from the supplied record without outside knowledge.

## Schema

Each line of `samples.jsonl` is one JSON object:

- `sample_id`: stable unique identifier;
- `category`: one of the eight behavior families;
- `domain`: scenario domain;
- `input`: chat-formatted task messages;
- `criteria`: exhaustive material conditions for a passing answer;
- `ideal`: one human-authored reference answer for review, not the sole permitted
  wording.

Each line of `grader_validation.jsonl` contains:

- `validation_id`, `category`, `input`, and `criteria`;
- `completion`: a candidate answer to grade;
- `choice`: human label `Y` for fully passing or `N` for materially failing.

The validation set contains eight positive and eight negative candidates, covering
all eight categories. Negative examples include subtle omissions, obsolete-state
resurrection, silent conflict resolution, cross-scope contamination, embedded
instruction following, priority mistakes, and unjustified certainty.

## Grading policy

The model-graded rubric assigns `Y` only if every material requirement in
`criteria` is satisfied. Any omitted prerequisite, inactive or out-of-scope fact,
hidden unresolved conflict, followed embedded instruction, size-limit violation,
or unsupported certainty receives `N`.

This strict all-material-requirements rule is intentional: in operational context
reconstruction, a fluent answer with one missing safety gate or one resurrected
decision can still be unusable.

The 16 human labels enable the upstream `metaeval: true` path. They do not prove
that a particular grader model is reliable; the meta-eval must be run and inspected
with the chosen grader.

## Data rights, privacy, and security

All text is original and fictional. It contains no personal information,
confidential records, copyrighted dataset excerpts, real secrets, raw production
traffic, or paid-call payloads. Credential-like requests inside a few cases are
plain fictional attack strings and contain no usable values.

The evaluator prompt explicitly delimits the task record, candidate answer, and
criterion as untrusted data. This reduces accidental instruction following by the
grader but is not a general prompt-injection guarantee.

If submitted to `openai/evals`, the contribution is intended to use the same MIT
license as that repository. The human contributor must confirm this license and
data-rights declaration in the upstream pull-request template.

## Known limitations

- The dataset is synthetic and English-only.
- Twenty-four cases are useful for an initial focused eval but do not represent the
  distribution of all long-running workflows.
- The contexts are long enough to require careful state reconstruction, but they do
  not stress maximum provider context windows.
- Model-based grading can share biases with the evaluated model and must be checked
  with the labeled meta-eval and human spot review.
- Exact bullet and word constraints make some cases intentionally strict.
- The eval does not compare retrieval systems, storage systems, summarizers, or
  orchestration products.
- A high score does not establish permanent memory, safe autonomous action, or
  correctness outside the supplied record.

## Recommended reporting

Report at minimum:

- overall accuracy;
- per-category accuracy;
- chosen evaluated model and grader model;
- prompt/configuration version;
- invalid-grader-output rate;
- meta-eval accuracy;
- every failed sample identifier; and
- the number of repeated runs if nondeterministic settings are used.

Do not claim broad model improvement from a single run or from structural file
validation alone.
