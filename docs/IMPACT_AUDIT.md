# Acheon impact and evidence audit

## Technical conclusion

**Conditional go.** Acheon has a credible path to major long-term impact as a
deterministic, auditable context-governance layer between retrieval systems and
models. The current release is a strong research alpha. It has not yet demonstrated
a revolutionary or general improvement in model behavior.

## Why the project is worth continuing

The problem is important: long-running agents need more than semantic similarity.
They must preserve current constraints, reject revoked or expired material, keep
projects isolated, retain required evidence, disclose unresolved conflicts, and
explain why any record was or was not placed into a bounded model context.

Acheon's most defensible contribution is the integration of these behaviors into a
single deterministic packet contract:

- lifecycle and scope gates before ranking;
- protected records and dependency-complete selection;
- explicit supersession, revocation, expiry, and conflict visibility;
- typed trust and provenance metadata;
- a reason code for every selected and omitted record;
- a canonical packet digest bound to the verified source state;
- append-oriented revisions and a tamper-evident local audit chain;
- an offline, model-independent core.

The individual ranking components are established retrieval methods. The credible
innovation claim is therefore system-level governance and audit integration, not a
new foundation-model architecture or a globally first memory algorithm.

## Verified evidence

As of 2026-07-22:

- 61 unit and integration tests pass;
- formatting, static checks, and release verification pass;
- the fixed synthetic workload has 240 cases in six named configurations, 64
  records per case, and a common 640 estimated-unit final-packet budget;
- Acheon records 1.000 Recall@budget, 0.857 precision, zero forbidden inclusion,
  zero budget violations, and deterministic selection on that workload;
- the release retains 1,731 structured failures from controls and ablations;
- one GPT-5.6 Sol Responses API smoke test completed and captured a provider
  receipt without publishing the answer text or private payload.
- one separate 24-sample GPT-5.6 Sol context-integrity run recorded 16/16 agreement
  on labeled grader checks, 23/24 automated passes, no invalid grader output, and
  one human-reproduced omission of required traceability details.

## What the evidence does not establish

The 240 cases share the same six-relevant/four-forbidden role structure. Gold labels
and system metadata were designed together, so the workload strongly favors the
declared governance contract. Most per-case differences are constant, making the
very tight bootstrap intervals measurements of template repetition rather than
real-world generalization.

The controls are chronological prefix, recent tail, and lexical top-k. They do not
include metadata-aware BM25, embedding or hybrid retrieval, summarization,
provider compaction, full long context, or mature memory systems. Disabling rank
fusion causes no measured change on the current generator.

The live adapter observation proves only that the integration completed. The
standalone context-integrity run tests the model directly on supplied records; it
does not run Acheon, compare a baseline, retain raw answers, use an independent
grader, or repeat the full experiment. It therefore establishes one reproducible
failure candidate for the public Eval, not an Acheon-attributable model improvement,
native context expansion, permanent memory, or a paradigm-level result.

## Competitive context

External memory, hierarchical context management, lifecycle-aware memory,
dependency-aware context construction, conflict handling, and auditable symbolic
memory all have published precedents. Relevant primary sources include RAG,
MemGPT, MemOS, ContextWeaver, MOSS, HiMem, LongMemEval, and LoCoMo. Acheon's
positioning must acknowledge those precedents and focus on its inspectable
governance contract and reproducible implementation.

- RAG: https://arxiv.org/abs/2005.11401

## Evidence gate for stronger claims

Before claiming observed model-behavior improvement, the project should complete:

1. held-out tasks not generated by the current workload template;
2. same-model and equal-token comparisons with strong retrieval, summary, and
   full-context baselines;
3. repeated runs with pre-registered structured rubrics;
4. answer correctness, constraint retention, stale-information use, dependency
   completeness, scope leakage, conflict disclosure, tokens, latency, and cost;
5. a separate write/update/revoke/reject memory evaluation;
6. multiple models plus at least one independent replication or real pilot.

Only stable downstream advantages would justify the phrase "observed model-behavior
improvement." Independent replication and sustained real-workflow gains would be
needed before discussing paradigm-level impact.

## Decision

Proceed with an OpenAI Evals contribution, Developer Showcase package, Developer
Community technical post, and a sanitized feedback protocol. Public communication
must call the project an alpha governed-context compiler with credible high-impact
potential and keep revolutionary impact as a testable hypothesis.
